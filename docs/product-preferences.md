# 产品设计偏好

本文档归纳期货决策系统的产品与 UI 取向，供后续改版、加数据模块、写知识型 Skill 时对齐。源于 2026-06 品种决策页反馈；2026-06 补充决策闭环与产品北极星。

## 北极星（为什么存在）

主流 APP 的问题不是信息少，而是 **信息散、且不敢帮用户组织判断**（合规上只能陈列事实，不能形成观点链）。

炒期货时，交易者真正要回答的是 **三个问题**：

1. **现在值不值得占用注意力？**（筛选）
2. **若值得，当前主导逻辑是什么，盘面是否还在配合？**（判断）
3. **若要做，什么条件下动手、什么条件下认错离场？**（行动边界）

> **帮你在有限时间里，从「信息过载」走到「有假设、有证据、有边界」——像值班研究员交班，不像股评，也不像行情终端。**

系统提供研判与结构化信号，**不替代**用户下单；读完应能自己执行最后一步，而不是被系统「带着走」。

## 产品定位（一句话）

> **快速扫品种、抓异常、看结论的决策面板** — 不是行情终端，也不是研报阅读器。

用户要在较短时间内走完上述三个问题，而不是读完所有可计算字段。

| 停留时间 | 用户应得到 |
|----------|------------|
| **约 10 秒** | 值不值得盯、偏多偏空、关注分 |
| **约 1 分钟** | 状态 + 命中证据 + 待验 / 失效 |
| **需要时** | 下钻量价、结构、宏观、persona 分歧 |

## 决策闭环（如何形成交易决策）

成熟决策很少来自单一指标，而是按下面 **六步** 推进。`insight`、Playbook、`core_hits` / `gaps` 均应对齐此链，**总览只露出链上的当前节点**，而非平铺全部模块。

```
状态 → 主导叙事 → 一致/分歧 → 缺口 → 触发 → 失效
```

| 步骤 | 用户在问 | 系统中的对应 |
|------|----------|--------------|
| **1. 状态** | 现在处于什么行情阶段？ | `state` · `level`、综合提炼首句 |
| **2. 主导叙事** | 当前谁在给价？逻辑是否仍成立？ | Playbook 各层 + `persona_*` 世界观 |
| **3. 一致/分歧** | 哪些维度同向，哪些在打架？ | 方向矩阵、板块共振；未来 persona 一致/分歧 |
| **4. 缺口** | 还缺什么证据才更敢动手？ | `gaps`、缺数据直说 |
| **5. 触发** | 什么价位/事件出现，从「观察」变「可行动」？ | 关键位、`confirmation`、经济日历 |
| **6. 失效** | 什么出现说明判断错了？ | `invalidation`、待验中的否定条件 |

**「观点」在本产品中的含义**：不是一句多空口号，而是在某种框架下，盘面 **支持/不支持** 哪条逻辑，还 **缺什么**、**错在哪会打脸**。persona 提供结构化立场；合规上不替用户喊单。

## 注意力漏斗（我们在关注什么）

注意力按 **决策顺序** 漏斗下行，而非按「数据类型」平铺。K 线、指标是 **叙事成立后的确认证据**，不是第一层。

```
背景（宏观 + 板块）→ 品种故事 → 结构（库存/期限/持仓）→ 盘面确认 → 催化剂 → 行动边界
```

| 层次 | 交易者心里在问 | 项目中的对应 |
|------|----------------|--------------|
| **0. 筛选** | 今天盯哪几个？ | 大盘首页、关注分、板块同向 |
| **1. 背景** | 风险偏好、利率、日历窗口 | `macro_regime`、宏观条 |
| **2. 故事** | 供需、政策、季节还是资金在定价？ | Playbook + persona |
| **3. 结构** | 紧还是松？近端逼仓还是远端过剩？ | 仓单、期限结构、持仓排名 |
| **4. 确认** | 故事对了，盘面认不认？ | 量价、矩阵、关键位、形态 |
| **5. 催化剂** | 什么会改主意？ | 经济日历、Playbook 确认/证伪 |
| **6. 行动** | 做不做、错了怎么办 | 关键位、待验/失效；**下单仍为用户** |

与早期「期货日报」草图的对应：

| 草图字段 | 产品语言 |
|----------|----------|
| 值得关注的品种 | 筛选层 |
| 为什么值得关注（宏观/行业/基本面/消息/技术） | 故事 + 证据链（非五个平铺栏目） |
| 影响程度 / 影响时间 / 行情消化力度 | 催化剂三维：冲击多大、持续多久、价格是否已反映 |

## 与主流 APP 的差异

| 主流 APP | 本产品 |
|----------|--------|
| 信息平铺 | 按 **决策漏斗** 分层；总览只答三个灵魂问题 |
| 无立场 | **多 persona 结构化立场** + 一致/分歧（不和稀泥） |
| 实时 tick 焦虑 | **日/周/月节奏**，交班式简报 |
| 指标大全 | **与当前叙事相关的证据** |
| 黑盒信号 | **可核对的事实 + 可执行的待验/失效** |

## 使用节奏与核心价值

产品 **不是** 鼓励持续盯 K 线、被盘面带着走。

- **K 线默认折叠**：一直看 K 线未必能看出更多结论，反而容易情绪化；K 线仅作需要时的辅助验证，不是主视线。
- **日度几次刷新即可**：关注核心信号，每天打开看板刷新几次，掌握当前市场状态就够。
- **周期以日/周/月为主**：不做高频盯盘，不提供 1 小时线维度（矩阵、K 线、信号分组均不含小时）。
- **核心价值**：① 知道市场 **现在处于什么状态**；② 理解状态背后 **涨跌的原因**（综合提炼、矩阵、关键位等结构化结论，而非实时 tick 行情）。

改版时优先强化「状态 + 原因」的可扫读表达，避免把产品做成行情终端或「必须一直开着看」的工具。

### 改版方法：提炼思想，统一落地

反馈往往指向 **一条原则**（如「辅助文字别占视线」），不是某一个词。改 UI 时：

1. 先写出原则（见 [ui-design-spec.md §1.2](./ui-design-spec.md#12-辅助信息全局一次行内只留差异) 等节）
2. 列出 **所有同类触点** 一并改
3. 避免只删单个文案留下另一种重复（删「查看走势图」却每行仍写「180日」即反例）

待 **一轮统一改版** 的项见 `ui-design-spec.md` §12.1。

## 核心壁垒（三层价值链）

项目随功能丰富，壁垒不在「能看行情」，而在 **帮决策者想清楚、敢下单**：

```
多维数据 → 专家知识库 → 精简呈现 → 读者做最终决策
```

| 层级 | 做什么 | 项目中的对应 |
|------|--------|--------------|
| **1. 多维数据** | 价量、技术、期限结构、板块共振、宏观变量等可交叉验证的证据 | 天勤行情、大盘首页、`src/analyzers/` 数据层 |
| **2. 专家知识库** | 在繁杂数据上抽丝剥茧，把研究员思维蒸馏成可复用的判断与信号 | Playbook、`research_lens`、`insight`、状态/级别/方向判定 |
| **3. 精简呈现** | 条理清晰、不重复、默认可扫读；读者像 **最终决策者**，而非被动看客 | 综合提炼、决策解读、矩阵、关键位；K 线/详表默认收起 |

**边界**：系统提供研判与信号，**不替代**用户下单；用户读完应能自己执行最终操作，而不是被系统「带着走」。

加新能力时自问：

1. 这是在帮用户 **筛品种、建叙事、找证据、定边界** 里的哪一步？答不上来则可能是数据囤积。
2. 是在加厚数据维度、强化知识蒸馏，还是优化决策可读性？三者缺一，壁垒都会变薄。

## 知识库设计约束（2026-06 确认）

> 材料依赖 **人工更新**；研报里的 **数字在入库时已过期**。因此 **不做 RAG 主库**，只蒸馏 **研究方法论**。

| 存什么 | 不存什么 |
|--------|----------|
| 变量名、检查顺序、因果链、证伪条件 | 研报中的库存/价格/目标价、当周多空口号 |
| 当前事实与信号 | 从 PDF 向量检索后直接生成结论 |

```text
Playbook（方法论） + Analyzer（最新数字） + Catalog（溯源）
parsed/*.md 仅用于蒸馏阅读，不参与 runtime 检索喊单
```

完整规范：[knowledge-distillation.md](./knowledge-distillation.md)。加新材料、新 skill 时先过本条；若方案依赖「把 PDF 扔进去就能问」或「引用报告里的数」，则与本产品方向不符。

## 解读话术风格

决策解读、综合提炼、走势摘要、矩阵解释等 **面向用户的文案**，统一气质：

> **平静 · 明确 · 务实**

像值班研究员交班记录，不是股评喊单，也不是研报摘抄。

### 平静

- 陈述事实与状态，不煽动情绪，不用惊叹、夸张副词。
- 允许不确定性，用「待验证」「需观察」「若…则…」代替「必然」「务必」「重磅」。
- 不用感叹号；少反问、少「注意！」「警惕！」式标题党。

### 明确

- 一句说清 **状态 + 级别**（或大盘一句主判断）；命中/缺项/待验分条，不拼成长段落。
- 每条命中尽量 **信号名 + 短解释**；避免「在…背景下考虑到…」从句套从句。
- 方向、关注分用标签展示，正文不重复喊「偏多」「高关注」。

### 务实

- 写 **能核对** 的盘面事实（价位、周期、指标名、持仓/量变化），少空泛宏观感慨。
- 「待验」写可执行条件（如「站稳 3800」「跌破 3720 失效」），不写「持续关注后续表现」。
- 缺数据直说「仓单未接入」「宏观未纳入本次」，不糊成「尚待进一步研究」。

### 示例

| 避免 | 更好 |
|------|------|
| 重磅！趋势引爆，务必紧盯多头机会 | 日线趋势上行；小时结构滞涨，需观察确认位 |
| 市场风起云涌，建议积极布局 | 板块黑色 3 品种同向；关注分 72 |
| 在看到更多信号之前请保持谨慎 | 若跌破 3720，当前偏多判断失效 |
| KDJ 高位死叉意味着短线惨烈回调一触即发 | 周线 KDJ 死叉；短线回落风险上升 |

**落点**：`insight` / `summary` / `synthesis` 产出、`global_market` 走势解读、Playbook 蒸馏与前端 `renderInsight` 展示均对齐本表。新增 skill 写 `interpretation`、`summary` 时先过一遍本节。

## 核心原则

### 1. 信息分层，同一结论只说一次

- 状态判断（如「趋势上行后的反转预警 · 中级行情候选」）集中在 **综合提炼** 等主叙事位输出。
- 左侧列表、Hero、矩阵等位置 **不重复** 同一套状态/级别文案。
- 方向结论用 **标签/色块**（多/空/中立、关注分）表达即可，避免再用长句复述（如矩阵标题下不再写「整体偏多」）。

**检查**：新增字段前问——这条信息和综合提炼是否重复？重复则删、合并或降级为可展开细节。

**决策解读排版**：标题行只写「状态 · 级别」；方向与关注分仅在栏头展示。命中/缺项/待验用**分行列表**，信号名与解释分两行，禁止用 `·` 拼成长段落。

### 2. 主次分明：核心默认可见，辅助默认收起

- **首屏优先**：综合提炼、方向矩阵、关键位卡片、核心指标。
- **参考项默认收起**：K 线、长表格、全量信号列表等「想看再展开」。
- K 线属于辅助验证，不是第一视线；占高度的大块内容应可折叠，且 **默认折叠**。

### 3. 控制认知负荷：能扫读，不必读完

- 默认展示 **够做决策的最小集**，而非系统能算出的全部。
- 详表、信号列表：**限量展示**（如每周期 4 条、表格 6 行），超出用「另有 N 条未展示」提示。
- 卡片区已表达的信息（如最近支撑/压力/确认位），**不再**用同主题全量详表重复铺满。

### 4. 布局符合工作流：索引与深读分离

- 左侧品种列表 = **索引**，右侧详情 = **深读**。
- 两侧 **独立滚动**；切换品种时列表滚动位置应稳定。
- 避免整页从上滑到下、列表与详情互相拖累。

## 更想要 vs 不太想要

| 更想要 | 不太想要 |
|--------|----------|
| 综合研判、方向矩阵、关键位摘要 | 重复的状态描述、冗长统计句 |
| 可折叠 K 线、精简表格 | 默认展开大图、全量详表 |
| 标签化结论（多/空/关注分） | 同一结论多处文字复述 |
| 双栏独立滚动、稳定导航 | 整页滚动、信息挤满首屏 |
| 冷静简报感、低饱和装饰 | 鲜红圆点分数、亮蓝选中底、五彩徽章 |
| 稀疏数据并排、少下滑 | 仅几行却全宽占屏、表格 `min-width` 撑满空白 |
| 辅助信息全局说一次，行内只留差异 | 每行重复「查看走势图」「180日」等相同提示 |
| 模块标题即足够 | 标题旁装饰 chip（教学说明、项数、「快速抓住重点」） |
| 平静、明确、务实的交班式话术 | 股评腔、喊单、夸张副词、空泛正确的长句 |

## 改版检查清单

加新模块或改 UI 时自问：

1. 该模块落在 **决策闭环六步** 的哪一步？总览是否需要展示它？
2. 这条信息和「综合提炼」是否重复？
3. 这是决策必需，还是查阅可选？可选则默认折叠。
4. 用户 10 秒内能扫完吗？不能则减量或分层。
5. 列表与详情是否各自滚动、互不拖累？
6. 后端 skill 输出的 summary 与前端展示是否在说同一件事？
7. 解读文案是否平静、明确、务实？有无股评腔或不可验证的空话？

### 改版验收清单（30 秒，2026-06 沉淀）

动 UI、分析周期、图表或顶栏前，逐项过一遍（细节与代码落点见 [ui-design-spec.md §12.2](./ui-design-spec.md#122-改版验收清单实操)）：

- [ ] 这条信息和「综合提炼」重复了吗？
- [ ] 10 秒内能扫完吗？不能就减量或折叠。
- [ ] 列表和详情是否各自滚动？
- [ ] 信号/点位是否还有 **小时级**？关键位是否 **优先日线**？
- [ ] 同一模块里 **卡片与全量表** 是否重复？
- [ ] 表格是否限量 + 正确提示（「另有 N 行未展开」，勿套关键位话术）？
- [ ] 单折线图是否复用 `drawMiniLine`，而非手写 canvas？
- [ ] 涨跌色是否 **红涨绿跌**（含偏多标签、K 线）？
- [ ] K 线是否 **默认折叠**？
- [ ] 顶栏是否 **品牌 / 导航 / 操作** 三层分离？
- [ ] 改 `src/` 后是否 **重启 8011**？用户是否需要点 **「刷新数据」**（不只强刷页面）？

**改版前三问**（动代码前）：

1. 落在决策闭环哪一步？总览要不要露？
2. 默认周期是日/周吗？旧缓存要不要 `sanitize`？
3. 有没有现成 `render*` / `drawMiniLine` 可复用，避免再写一套？

## 已落地的示例（品种决策页）

- 左侧列表去掉 `state · level`，仅保留板块与合约。
- 品种顶栏 **合并一行**：左（名称+合约+标签），右（最新价/涨跌幅/保证金/日均波动）；去掉独立「夜盘」指标格，夜盘状态保留在标签。
- Hero 去掉与综合提炼重复的 `state · level`。
- 方向矩阵去掉 **小时线** 维度与「整体偏多/偏空」副标题，保留 bias 标签；解读区展示 `judgment_note`，去掉重复关注分 pill。
- 量价/技术/形态信号统一 `renderSignalCardGrid`；无 `period` 的信号不显示周期 chip。
- 分析周期 **仅日/周**（后端 + 缓存剥 hour + 前端二次过滤）；关键点位脚注应为「日线」。
- K 线图单层折叠、默认收起、红涨绿跌；切换品种时重置为收起。
- 移除「关键点位详表」全表；卡片保留最近关键位；其他详表与信号列表限量展示。
- **资金结构·期限结构**：跨期价差用卡片、合约链单表（≤5 行）、折线图走大盘同款 `drawMiniLine`。
- **顶栏**：左品牌 · 中页面切换（主任务）· 右刷新（描边次按钮）。

## 关注分与打分体系（暂定，待重塑）

**2026-06 共识**：当前规则 **暂时保留**，不在近期改版；随数据维度增加（产业、期限、持仓、板块共振、日历等接入并稳定后），**打分体系必然要重塑**。

### 现行逻辑（`src/analysis.py`）

1. **先状态、后分数、方向独立** — `classify_state` → `score_state`；`direction` 由状态文案映射（偏多/偏空/中性），与分数分开。
2. **关注分 = 注意力优先级，不是多空强度，也不等于当日涨跌幅。**
3. **状态基础分**（约从 35 起）：突破 +25、趋势 +18、反转预警 +20、波动扩张 +16、收敛 +10；中性无信号封顶约 45。
4. **叠加项**：成交量 60 日分位、ATR 分位、持仓变化、日线路径（连续上/下行）、周线与日线同向等。
5. **列表排序**：按关注分降序；**偏多 top 与偏空 top 均可高分**（例如放量向下突破：偏空 + 高关注）。

### 已知局限（重塑时需解决）

| 局限 | 说明 |
|------|------|
| 维度单一 | 几乎只反映日线量价结构，产业/资金/宏观未进入分数 |
| 多空不对称未建模 | 高分不区分「值得做多」与「值得做空」，需结合方向标签理解 |
| 与决策闭环未完全挂钩 | `level`、矩阵、Playbook 命中未反哺分数 |
| 扩展债务 | 每加一个 analyzer，若不进评分公式，列表优先级会与「信息丰富度」脱节 |

### 重塑方向（记录备查，非近期任务）

- 明确分数语义：继续「关注度」还是拆成「活跃度 + 方向置信」双轨。
- 将已接入且稳定的维度（板块共振、期限、持仓、基差等）纳入 **可解释加权**，避免黑箱。
- 总览筛选与列表排序规则一并调整，并写清与 `state` / `level` / `direction` 的分工。
- 改版时同步更新交接文档、侧栏图例与 `judgment_note` 话术。

**实现前必读**：本段 + `analysis.py` 中 `classify_state` / `score_state` / `classify_level`；重塑时优先写设计说明再动代码。
